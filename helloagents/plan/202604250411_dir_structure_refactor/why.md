# 变更提案: 目录分层重构与兼容适配

## 需求背景
当前项目将源码、运行脚本、测试数据、测试入口、报告产物和副本文件混放在根目录中。该结构已经导致以下问题:
- 根目录文件职责混杂，难以区分核心模块、实验脚本、兼容入口和归档内容。
- 现有模块大量使用根目录绝对导入，任何直接移动都可能破坏 `python main.py unit`、`pytest` 和脚本间互调。
- `reports/` 中存在重复嵌套的 `reports/reports/` 结构，旧产物与当前产物混杂，难以追溯。
- `* copy.py`、`README copy.md`、`pyproject copy.toml`、`reports copy/` 属于历史副本，需保留但不应继续污染主工作区。

## 变更内容
1. 建立 `src/`、`scripts/`、`tests/`、`reports/`、`archive/` 分层结构。
2. 将核心实现迁入 `src/`，将可执行实验与统一入口迁入 `scripts/`。
3. 在根目录保留兼容薄封装，继续支持 `python main.py unit`、现有测试命令和旧式模块导入。
4. 归档所有 `copy` 副本和旧 `reports copy/` 目录，避免污染正式目录。
5. 规范 `reports/` 产出路径，消除可避免的重复嵌套层级。

## 影响范围
- **模块:** 统一入口、核心 DSRA 模块、实验脚本、报告工具、测试导入链
- **文件:** `main.py`、`report_utils.py`、`dsra_layer.py`、`dsra_model.py`、`toy_task_associative_recall.py`、`json_retrieval_test.py`、`attention_family_benchmark.py`、`ablation_study.py`、`benchmark_complexity.py`、`needle_in_haystack_test.py`
- **API:** 无外部 API 变化
- **数据:** 报告目录与归档目录布局调整

## 核心场景

### 需求: 分层目录重构
**模块:** 目录结构
将源码、脚本、测试、报告和归档内容按职责拆分到稳定目录。

#### 场景: 新开发者浏览仓库
开发者进入项目根目录后，希望快速定位源码、脚本和测试。
- 根目录仅保留少量兼容入口和工程配置。
- 业务源码集中在 `src/`。
- 可执行脚本集中在 `scripts/`。

### 需求: 兼容旧调用方式
**模块:** 兼容适配层
在分层后仍然保持原有 CLI 调用、测试命令和脚本导入可用。

#### 场景: 继续执行历史命令
用户运行 `python main.py unit` 或旧测试导入 `from json_retrieval_test import ...`。
- 命令无需修改即可成功执行。
- 导入路径通过兼容薄封装继续工作。

### 需求: 规范报告与归档
**模块:** 报告与归档
将正式报告路径与历史副本分离，并简化重复层级。

#### 场景: 查看本次实验产物
用户查看 `reports/` 中的最新输出。
- 正式输出始终落在 `reports/` 主树下。
- 历史副本保存在 `archive/`，不与正式报告混淆。

## 风险评估
- **风险:** 直接移动模块后，旧式绝对导入可能全部失效。
- **缓解:** 先迁移正式实现，再在根目录建立一对一兼容薄封装，并用测试与命令验证。
- **风险:** 报告目录重构可能影响既有写入路径。
- **缓解:** 保持 `ensure_reports_dir()` 的对外行为稳定，仅规范调用基准目录与归档位置。
